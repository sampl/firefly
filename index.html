Making websites is still too hard.

Bulding a static site is pretty easy--but learning how to let users log in, build and properly validate form input, and add search is a whole different ballgame. Many people just give up.

Luckily, some new tools are making all this a lot easier. When combined, they make a pretty compelling package.

That's what Firefly is--a couple of modern web tools put together to help you build real web apps. Think of it as a tiny alternative to Rails for people writing MVPs. Building something in Firefly is crazy fast--and when your project takes off, it's easy to extend with whatever else you need.

I hope it helps you make something great!

Warning

Firefly makes web development radically simple by not using backend servers, relational databases, unit tests, and other sane best-practices for enterprise codebases.

Philosophy

Cautiously Modern

Firefly takes advantage of some relatively new ideas in web development in order to make the code as small as possible.

  • Modern Javascript - Firely uses modern Javascript and the huge JS ecosystem, with one major caveat: we don't go bleeding-edge. All libraries and services we use are well-established, used in production by major companies and will be supported for years to come.
  • Heavy front-end - Firefly uses the front-end as much as possible. This lets us have a single codebase for most of our app logic.
  • Serverless backend - What Firefly can't do on the client or with a 3rd party service, it does using Firebase Functions, which require no uptime monitoring or special deployment.

Radically Simple

Firefly's goal is NOT to be a toy for experienced engineers--it's to empower casual developers with other skills build working products as quickly as possible. That's why Firefly should not just be a great tool, but also a great teacher.

  • Less code - Lots of app boilerplates are daunting, but Firefly has as little code as possible--40 files at last count. The less code your app has, the less you have to understand and maintain.
  • Boilerplate - Firefly is just a demo app, not a library or framework. There are no scaffolds and no CLI. To add more models and views, just copy-and-paste the ones that are already there.
  • Extensible - Making it easy to add code means we don't have to include extra crap out-of-the-box. Firefly only has login, form validation, and search--everything else you add on your own.
  • Services - Instead of including code for complicated things like authentication and search, Firefly outsources all that to outside services. This increases the chances of vendor lock-in, but vastly reduces the amount of complex code for beginners.

How it works

Firefly is a single-page front-end web app boilerplate. Models have clear separation from Firefly views (Kind of like MVC, without the "C").

If you're used to working with jQuery, Rails, or Express, Firefly is going to feel very different. There are a few concepts that are important to understand before you dive in.

This not only makes it easier to, but easier to swap out smaller pieces of your app as new packages become available.

The Post model and views are just as an example copy it to create more collections and views or rename it to reflect the main collection in your app.

As you read, follow along with the code on github.

Models

The database

Firefly uses the NoSQL Firebase Real-time Database for persisting data. Firebase is basically a big JSON file in the cloud that you can read/write to directly from a users web browser (no server required). Before you go much further, you should read the Firebase Real-time Database Guide.

Since Google Firebase is basically just JSON, it has no schema, no tables, and no collections. The convention in Firefly, however, is to use the top level of the database as a collection, the second level for unique keys for items in that collection, and the third level for the attributes of each item (like a row). For example:

{
  "dogs": {
    "1": {
      "name": "spot",
      "age": "4",
    },
    "2": {
      "name": "fido",
      ...
    },
    "3": {...},
  },
  "cats": {...}
}

Firefly models

You shouldn't have to write to the database directly. Firefly makes it easy to add, read, and remove items with Javascript objects called models.

Firefly models live in the models folder. You can write a Firefly model however you like, but the default models extend a Supermodel class that has a lot of helper functions for adding, querying, and deleting collection items.

Most models simply modify and return the Firebase promise on the Firebase query library. This makes error handling in the views very simple. However it does require a bit of knowledge about how promises work.

The User model

The User model is special--it has custom methods for signing in and out with Firebase auth.

We use Firebase Authentication for logging users in, etc.

Relational data

Handling data that "relates" to other data is one of the most difficult parts of writing models. For example, how can you mark a post as "liked"--and see all the posts a user liked--without saving that data in two places?

Firebase handles this much differently than most apps. Each collection can be directly manipulated by the clients. Each collection has some attributes that are only updated by Firebase server cloud functions.

For example: when you like a post in Firefly, Firebase adds a new "like" to the post_likes collection. Then, a Firebase function adds 1 to the post's _like_count attribute so we can query and sort posts by the number of likes they have.

Loop to query lots of results--fairly fast because Firebase batches db requests

Securing your data

Because clients can write data directly into your database, you need to write some security rules to keep bad guys from deleting data or viewing private data. This video does a great job explaining how these rules work.

Views

Component-based views

Views in Firefly are built with React, which keeps a component's controller logic, templates, and even CSS all in a single file. If you're used to writing HTML, CSS, and JavaScript in different files, this will feel weird at first. Stick with it--writing your UI using all-on-one components will feel like a revelation soon. Start by reading the official React guide.

Routing

The router is the part of the app that shows different pages based on the URL. We use the popular React Router.

Listening for updates

Many React apps use something like Redux to for state management (which sort of a fancy way of saying "keep my views up-to-date when my data changes"). Many developers agree that Redux is overkill for a lot of simple apps.

Instead, Firefly uses the much-simpler eventemitter3 library to let models announce when they've changed, and let views listen for those changes so they know to re-render.

Other stuff

Firefly uses Create React App and Firebase command-line tools to take care of transpiling, bundling, running tests, and deployment. You don't have to write or manage any configuration for this to work, just run npm start normally.

Bundling

Modern Javascript requires transpiling code to ensure that your app works on all browsers. Under-the-hood, Create React App uses Webpack to bundle all your files into a single app.js. It starts at src/app.js, finds all the files that it imports (and the ones they import, etc) and bundles them all up into a single app.js.

Hosting

We use Firebase hosting to host your app. Firebase hosting comes with HTTPS for free, and it's the same price as Google Cloud Platform object storage (very cheap). See Getting Started: Deploying for directions on deploying your app.

Getting Started

Quick start

  1. git clone https://github.com/sampl/firefly && cd firefly && npm install
  2. mv .firebaserc.example .firebaserc && mv env.development.example env.development && mv .env.production.example .env.production
  3. add your Firebase keys to .firebaserc and the .env files
  4. npm start

Before you begin

This tutorial uses Git for version control. If you're not already a Git user, you can learn about it here.

Download and install node if you haven't already.

Run npm install -g firebase-tools to install the Firebase CLI. You only need to do this once per computer.

Creating your app

Get the code

Firefly is just a boilerplate. To start a new Firefly project, we simply make a copy of Firefly.

  1. In your terminal, cd to the folder where you want your project to live.
  2. Clone this repo with git clone https://github.com/sampl/firefly.
  3. Go into the folder you just created with cd firefly. You can also rename the firefly folder to whatever you want.
  4. Run npm install to install all the packages Firefly needs.

You've got your own Firefly app! Now let's tell it how to find the database.

Create the database

  1. Go to the Firebase console and sign in with your Google account
  2. Create a new project and name it something like MyProjectName (Stage)
  3. Click "Add Firebase to your web app"
  4. You'll see a block of code--leave this window open, you'll need it in a minute

Connect to the database

Now let's connect our database to our app. The connection parameters are security secrets, so first we'll rename our environment files to keep them from getting committed into git (the new filenames are listed in .gitignore).

  1. First, rename .firebaserc.example to .firebaserc and replace the blank space next to "dev" with the projectId from the Firebase website.
  2. Rename .env.development.example to .env.development. Replace the blank spaces in that file with the codes from the Firebase website too.

Set up authentication

Before you can add any posts, you'll have to allow sign-in with Google. Firebase makes this very easy.

  1. Back in the Firebase console, click on "Authentication" in the sidebar on the left
  2. Click "Set up sign-in method"
  3. Click "Google", toggle the "Enable" switch, and hit "Save"

As you work

Start your app with npm start. You should see a basic web app in your browser! ðŸ˜Ž

Try logging in, then make sure you can create, like, edit, and delete posts.

Now go back to your text editor and try editing some files. When you save, the changes should automatically appear in the browser.

After each change, commit your work using git.

When you're ready to deploy your app to stage, run npm run stage.

Deploying

Each project you make using Firefly actually uses two Firebase projects--one for development and staging, and one for the live production site. We already made the development project, now we need to make one for production.

  1. Go back to Connecting to Firebase and repeat the steps, but call this project something like MyProjectName - LIVE
  2. Fill in the "production" value in .firebaserc
  3. Rename .env.production.example to .env.production and add the new Firebase connection parameters there too.

  4. Firefly is insecure by default. Before going live, secure your app by writing rules in database.rules.json

When it's time to go live, run npm run live!

Next Steps

Create React App also includes ESLint to keep you from making small mistakes, the Babel transpiler to make sure the code you write works on all browsers, and Jest to run automated tests on your code.
